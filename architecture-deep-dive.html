! song.title is a Firebase key

Renaming a song requires updating charts, readiness, discussions, moments, section_ratings, plus 4+ other paths. The songs_v2 migration tool handles this; manual title changes can orphan child data.

! song.readiness is the most cascading field

One readiness edit invalidates: focus engine, intelligence ranking, home dashboard cards, rehearsal recommendations, weak-songs sort, and GrooveMate's "next song" suggestion. Cache invalidation here is critical; if any consumer holds stale data, recommendations diverge.

✓ meta/members now triggers a Cloud Function

Shipped 2026-05-07: any write to bands/{slug}/meta/members/{key} fires mirrorMemberToIndex which updates members_index. The auth gate reads from members_index. Edits propagate automatically.

⚠ calendar_events and gigs dual-mirror

Documented dual-source. Mostly stable after Tier 1-3 audit closures. Stage-2 source-of-truth flip planned (P2.2 in optimization_plan.md) to make calendar_events canonical and delete the gigs node.

✓ build-version is single-source-of-truth now

Console banners, GLContext, knowledge resolver, and service worker cache all read from <meta name="build-version"> at runtime. Bumping the meta tag (one of 4 atomic sources) propagates everywhere — no more stale banners.

✓ Notes now flow through one entry point

Phase A (2026-05-09): chart, rehearsal, gig, personal_critique, and stem notes all write through GLNotes.write(songTitle, scope, text). Each scope adapter preserves the existing Firebase path and field shape, so legacy readers are untouched. Adding a new notes UI now means picking a scope, not inventing a new save path.

! shared_setlists is a new public-read node

Slug-based gig-pack share URLs (?setlist=<slug>) store the pack at shared_setlists/<slug> with public read. Slug = slugify(name + '-' + date); re-saving the same setlist updates the same slug (idempotent). Anyone with the URL can view, even unauth — Firebase rule explicitly grants public read on this path only.

🔥 Phase 9 is the headline cost

~50,000 lines of feature code parsed even if the user only visits one page. On Pierce's iPhone 4G, this phase alone is 600-900ms. P0.1 in the optimization plan: lazy-load feature pages by route.

! 800ms magic-number race

setTimeout(showPage, 800) in app.js delays initial render to give Firebase + OAuth time to settle. Too long on fast networks (visible blank flash); too short on slow ones. P0.2: replace with event-driven Promise.all.

✓ Boot watchdog at 5s

If no page is visible 5s after DOMContentLoaded, force-show home. Last-resort fallback that catches every edge case (failed Firebase init, failed GIS, race conditions).

! setInterval cleanup audit

Multiple setIntervals start in groovelinx_store.js: sync heartbeat, stale check, status badge timer. P0.3: audit cleanup paths to prevent leaks.

6. What to do with this

Three concrete uses for this document.

📋 Refactor planning

Before changing an engine: open §1, find the engine row, look at the "consumers" column. That list is your test surface. Don't ship a GLStore change without testing all 19 features.

🔍 Bug triage

Multi-screen bug? Find the affected screens in §1, look up which engines feed them. The intersection is where the bug likely lives. Single-screen bug? Check §4 to see if it's in a hot-spot file (>5k lines).

🚀 Performance tuning

Slow startup on a tester device? Use §3 boot sequence to map their performance trace to specific phases. Check §5 to confirm it's not a known race. Cross-reference §4 hot spots for refactor targets.

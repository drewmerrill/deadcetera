Key data flows

How a typical action travels through the system.

1 Sign-in (auth gate)

Browser → Google OAuth → token → Cloudflare Worker /oauth/userinfo → email → app.js _glCheckBandMembership → single read of members_index/{key} in RTDB → band slug. ~50 bytes payload.

2 Adding a band member

Drew edits bands/$slug/meta/members → RTDB raises event → Eventarc → Pub/Sub → mirrorMemberToIndex Cloud Function → writes members_index/{email}: slug.

3 Stem separation

Browser uploads source URL → Worker /stems/start → Modal spins T4 GPU → Demucs 6-stem split → uploads to R2 → stem URLs returned → Tone.js wires up the mixer.

4 Real-time data sync

Any device writes to RTDB over WebSocket → Firebase fanout → every other signed-in device receives the change → active page re-renders. <1s end-to-end.

5 SMS notification

Trigger fires (gig change) → Worker /sms/send with band SMS subscriptions → Twilio Messaging Service → A2P-routed carrier → bandmate's phone.

6 Calendar sync

Each member's Google Calendar → Worker /calendar/freebusy → RTDB calendar_events → band view. Outbound: GrooveLinx gigs → shared band cal via events.insert.

7 Auto-deploy

git push → GitHub webhook → Vercel rebuild → CDN propagation → live in ~30s. Service worker version bump triggers cache refresh on next reload.

8 GrooveMate suggestion

Stems lens detects 3+ loop reps → GLContext snapshot → heuristic engine → optionally Anthropic Claude via worker → suggestion card with Apply / Dismiss.

## 2. Core Architecture

### App shell + boot order
- **Boot file:** `app.js` (~14,700+ LOC).
- **Boot sequence** (app.js:700–920):
  1. `DOMContentLoaded` → `renderSongs()` paints empty shell from hardcoded `allSongs` (line 743).
  2. `initFirebaseOnly()` initializes Firebase + Drive auth (line 754).
  3. Firebase resolves → loads members + song library + DNA metadata.
  4. `glHeroCheck(isSignedIn)` gates: hero page if no session, home if signed in (line ~890).
- **Status:** LIVE / STABLE.
- **Risk:** ~6 KB of inline JS in `index.html` boot watchdog (lines 801–860) prevents blank-screen on a failed boot; no equivalent on `app-dev.js` divergence.

### Router / navigation
- **File:** `js/ui/navigation.js`.
- **Surface:** `showPage(pageKey)` is the single router. No route table — routing is implicit: every `#page-*` div in the DOM is a valid route, and `pageRenderers[pageKey]()` is called on transition. Failed renderer calls are silent.
- **Routes registered (~25):** home, songs, setlists, rehearsal, rehearsal-intel, playlists, practice, workbench, calendar, gigs, feed, pocketmeter, venues, finances, tuner, metronome, equipment, contacts, stageplot, ideas, social, admin, notifications, bestshot, stoner, help, hero.
- **System lock:** `_navSeq` counter guards all 7 `GL_PAGE_READY` assignments per CLAUDE.md §7a. Do not modify without explicit review.
- **Risk:** No central route registry. Any `#page-*` div with no registered renderer silently no-ops — orphan-page risk.

### Auth + dev bypass
- **OAuth client:** `firebase-service.js:49–57`. Scopes: `email profile calendar.events calendar.readonly drive.file`.
- **Membership gate:** Hard block (mode A) per memory `project_auth_gate_mode.md`. Switch to mode B (self-onboard) is gated on Drew's decision when ready for founding members of other bands.
- **`currentUserEmail`:** Persisted in `localStorage.deadcetera_google_email`. Set on sign-in (app.js:5730), cleared on sign-out (app.js:5811).
- **Dev bypass:** `?dev=true` URL param + `GLT` global → auto-signs in as `test@groovelinx.com` on test band slug `deadcetera-test`. Stripped in production unless `?dev=true` present (firebase-service.js:86–98).
- **Band slug resolution:** `localStorage.deadcetera_current_band` (default `deadcetera`). All paths route through `bandPath()` helper at firebase-service.js:104–106. Multi-band ready; no per-URL slug param.

### State layer (`GLStore`)
- **File:** `js/core/groovelinx_store.js` (~1,057 LOC).
- **Owns:** `ACTIVE_STATUSES` (canonical), `isActiveSong()`, `getNowFocus()`, focus cache, event emitter (`focusChanged`), and the shared store contract.
- **System lock #2:** `focusChanged` event model per CLAUDE.md §7b. Home, Songs, Rehearsal must subscribe to this event; new focus consumers must subscribe (not bypass with inline weak-song calculations).
- **System lock #4:** `ACTIVE_STATUSES` and `isActiveSong()` are the single definitions; do not create inline `{ prospect:1, learning:1, ... }` objects.

### Firebase service
- **File:** `js/core/firebase-service.js` (~1,579 LOC).
- **Owns:** Firebase init, multi-band routing, `loadBandDataFromDrive`, `saveBandDataToDrive`, SWR cache helpers (including the hardened `_glSafeCache` shipped 2026-05-11).
- **System lock #3:** Firebase error filter in `index.html` suppresses ONLY `firebaseio.com/.lp` long-poll noise — do not broaden.

### Service worker + version detection
- **File:** `service-worker.js`. Cache name: `groovelinx-20260512-232320`. Cache-first for shell.
- **Update detection:** `index.html:801–818` runs `r.update()` on every load. **No user-facing "update available" UI** exists.
- **Risk:** If a user keeps a tab open across deploys, they could run old JS + new HTML (a mismatch only the boot watchdog catches indirectly). No graceful upgrade prompt.

### Feature module count (gl-*.js)
- **86 total core modules** under `js/core/*.js` (memory had previously cited "28 gl-*.js" — that referred only to the May 2026 split set; total core count including older modules and the 4 rehearsal engines is 86). Verify and update memory.

## 9. Final Recommendation

### Top 10 stabilization priorities (ordered by impact-to-effort)
1. **Delete `home-dashboard-cc.js`** (safe, declared dead in HTML) — 30 sec, 23 KB win.
2. **Wire `version.json` → in-app update prompt** OR stop writing it. Either close L6+L11 together.
3. **Mechanically replace 7 inline `ACTIVE_STATUSES` shadows** with `GLStore.ACTIVE_STATUSES` references. Fixes L3, prevents future drift.
4. **Hide or finish `workbench.js`** — registered route to a stub is bad UX. Either MVP it or remove from nav.
5. **CI parity check on `app.js` ↔ `app-dev.js`** — or auto-generate one from the other. Fixes L4.
6. **Decide on the setlist player contract** — enforce or remove. Avoid half-states. Fixes L5.
7. **Audit #02 — Data Access** — produce a migration plan for the 33 direct Firebase reads (don't fix yet; just plan).
8. **Audit #03 — `localStorage` keys** — inventory all `deadcetera_*` / `gl_*` keys; identify orphans; migrate critical ones to `_glSafeCache`.
9. **Audit #04 — Module size + split candidates** — propose criteria for splitting files >5,000 LOC.
10. **Remove `_legacy_` schedule block shim** after one production data check. Closes L12.

### First 3 audits to run after this one
- **Audit #02 — Data Access Inventory:** every Firebase / Drive / Worker call, what reads vs. writes it, what caches it, what owns invalidation.
- **Audit #03 — `localStorage` & Cache Topology:** every persistent key, owner, expected TTL, schema. Identify orphan keys.
- **Audit #04 — Page-Level Coverage Map:** for each `showPage()` route, what's its job, what features does it depend on, what's its acceptance criterion. Will inform the "one job per screen" rule from memory.

### First 3 things NOT to touch yet
1. **The A2P frozen surfaces** (4 public files + SMS opt-in UI) until carrier approval (~2026-05-26 to 2026-06-02 window).
2. **The four CLAUDE.md system locks** until any audit specifically targets them.
3. **`gl-calendar-sync.js` decomposition** — it's stable now and serves the entire Google Calendar integration. Don't refactor without an audit dedicated to it.

### Immediate red flags (require Drew + ChatGPT review)
- **🚩 `app.js` ↔ `app-dev.js` drift is invisible.** Today's session fixed 3 bugs in `app.js`; the same fixes were *not* explicitly mirrored to `app-dev.js`. Drew should decide: keep both files (with CI parity), or generate one from the other.
- **🚩 The "contract" pattern (player contract, GLStore centralization) is documented but not enforced.** Either tighten the contracts (lint / pre-commit hooks / runtime warnings on direct calls) or remove the contract layer.
- **🚩 33 direct Firebase reads is too many.** Even if each was intentional, the cumulative weight makes migration (e.g., songs_v2) painful. Should be a strategic priority before adding any new schema.
- **🚩 No "update available" UX** means users on iPhone keep stale JS for hours after a deploy. With the band doing live UAT, this risks running stale code at a gig.

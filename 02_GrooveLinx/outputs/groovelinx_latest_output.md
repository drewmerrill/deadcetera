## Feed Integration Logic

### Event Triggers

| Event | Feed Post |
|-------|-----------|
| New playlist created | "Rehearsal playlist ready — 15 songs" |
| Existing playlist updated | "Rehearsal playlist updated — 15 songs" |
| Re-sync after review | Same as update (fires through syncToSpotify) |

### Deduplication

Posts only fire when the track hash changes. If user taps "Sync" and the playlist is already up to date, no feed post (hash matches previous sync).

### Data Shape

```json
{
  "title": "Rehearsal playlist ready — 15 songs",
  "author": "Drew",
  "ts": "2026-03-24T...",
  "tag": "fyi",
  "link": "https://open.spotify.com/playlist/3cEYpj...",
  "targetType": "all",
  "_source": "playlist_sync",
  "_bundleType": "rehearsal"
}
```

- `tag: 'fyi'` — informational, no action required
- `link` — Spotify playlist URL (renders as clickable "🔗 Link" on feed card)
- `targetType: 'all'` — visible to entire band
- `_source` + `_bundleType` — metadata for future filtering/dedup

### Action Link

Each feed item includes the Spotify playlist URL as `link`. The existing feed card renderer shows this as a clickable "🔗 Link" that opens the playlist.

## Edge Cases

- **Already up to date:** Hash matches → no feed post (avoids spam)
- **Sync with failures:** Feed post still fires (matched songs are in the playlist)
- **Re-sync after review:** New hash → feed post fires as "updated"
- **Offline/Firebase down:** Feed post fails silently (console warn), sync still succeeds
- **No FeedActionState:** Author falls back to "GrooveLinx"

# GrooveLinx — Setlist Player v5: In-App Playback Overhaul

_Generated by Claude · 2026-03-24_

## Bug Report (Original)

- Open Grizz Fest setlist → click headphones
- Header shows "Wings Cafe 06/15/24" (stale prior session)
- Hangs on "Opening best version..."
- YouTube opens external YouTube search page
- Archive opens external Archive page
- Spotify button does nothing

## Root Causes Found (5 bugs)

### 1. Race Condition on Shared Closure State
Auto-resume fires on DOMContentLoaded if `gl_player_state` in localStorage is < 2 hours old. `_resumeFromState()` is async — while it awaits `_buildQueue()` (sequential network calls per song), user clicks headphones on a different setlist. Second `launch()` runs concurrently — both mutate the same `_queue`, `_setlistName`, `_currentIdx`. Queue gets corrupted, wrong header shows.

### 2. Index Mismatch in slPlaySetlist
`slPlaySetlist(idx)` re-fetched ALL setlists from Firebase and **re-sorted by date descending**. But `idx` was `_origIdx` from the **unsorted** raw Firebase array set at render time. After sorting, the index pointed to a different setlist entirely.

**Example**: Firebase returns [Wings Cafe, Grizz Fest, Practice]. Render gives `_origIdx`: Wings=0, Grizz=1. Click headphones on Grizz → `slPlaySetlist(1)`. Handler re-sorts by date → [Grizz, Wings, Practice]. Index 1 in sorted = Wings Cafe. Wrong setlist.

### 3. "Opening best version..." Hung Forever
`_showFallback()` set "Opening best version..." in the video container with no timeout, no terminal state, no way out. Text stayed forever.

### 4. Spotify Popup Blocked on Mobile
`_openSpotify()` was async. By the time `window.open()` ran after `await lb.resolveUrl()`, mobile Safari had lost the user gesture context. Popup silently blocked.

### 5. Pre-Resolution Blocked Overlay
`_buildQueue()` resolved YouTube IDs for ALL songs upfront via sequential Invidious API calls. With a 15-song setlist and both Invidious instances down, this blocked for 150+ seconds before the overlay even appeared.

## Fixes Applied

### Fix 1: Launch Token Guard (`_launchToken`)
Monotonic counter incremented on every `launch()`. After each `await`, token is checked — stale launches bail immediately. Prevents race conditions.

### Fix 2: Full Reset on Launch
`launch()` calls `fullClose()` + `clearResumeState()` before anything. Kills: previous YouTube player, overlay DOM, now-playing bar, queue, index, setlist identity, localStorage resume state.

### Fix 3: Index Mismatch — Remove Re-Sort
`slPlaySetlist()` no longer re-sorts. Uses raw unsorted Firebase array matching `_origIdx`.

### Fix 4: Lazy Queue Building (Instant Overlay)
`_buildQueue()` is now **synchronous** — collects song titles + band names + checks localStorage cache only. No network calls. Overlay appears immediately. YouTube IDs resolve lazily per-song when played.

### Fix 5: Parallel Search (7 Backends)
YouTube resolution now fires ALL backends simultaneously — first good result wins:
- 5 Invidious: vid.puffyan.us, invidious.fdn.fr, inv.nadeko.net, invidious.nerdvpn.de, invidious.privacyredirect.com
- 2 Piped: pipedapi.kavin.rocks, pipedapi.adminforge.de
- 4-second timeout per backend, 6-second overall timeout

### Fix 6: In-App Fallback (No External Links)
When auto-search fails, fallback shows **inside the player**:

| Option | Behavior |
|--------|----------|
| **Retry Search** | Clears cached ID, re-resolves across all 7 backends |
| **YouTube URL Paste** | Text input → extracts video ID → embeds via IFrame API |
| **Find on Spotify** | Searches via ListeningBundles → embeds Spotify track iframe |
| **Skip to next song** | Advances queue |

No external links by default. Player shell (setlist name, song, controls, up-next) stays intact regardless of source.

### Fix 7: Spotify Popup Fix
`window.open()` fires synchronously before any `await` (preserves gesture context). If a better Spotify URL resolves async, the already-open window is redirected.

### Fix 8: Spotify In-App Embed
"Find on Spotify" button searches via `ListeningBundles.resolveUrl()` or `searchSpotifyForSong()`, extracts track ID, embeds via `open.spotify.com/embed/track/{id}` iframe. Falls back to "Open Spotify Search" link only if no match.

## Architecture: Setlist Player v5

```
slPlaySetlist(idx)
  → loads setlist from Firebase (no re-sort, uses _origIdx)
  → SetlistPlayer.launch(setlistObj, name)

launch()
  → ++_launchToken (guards all async)
  → fullClose() + clearResumeState()
  → _buildQueue() — SYNCHRONOUS (no network)
  → _createOverlay(name) — shows immediately
  → _playCurrent()

_playCurrent()
  → if cached youtubeId + YT ready → _playYouTube()
  → else → _showSearching()

_showSearching()
  → _ensureYouTubeAPI() in parallel
  → _resolveYouTubeId() — races 7 backends
  → if found → _playYouTube()
  → if timeout/fail → _showFallback()

_showFallback()
  → Retry Search button
  → YouTube URL paste input + Play button
  → Find on Spotify button → _embedSpotify()
  → Skip to next song button
```

# GrooveLinx — Band Feed Workflow Hardening + Dashboard Status Filter

_Generated by Claude · 2026-03-24_

## Problem 1: Broken Back-to-Feed Navigation

**Root cause:** `_feedNavigatedFrom` was a volatile JS variable that reset on any page reload or when GLStore's `pageChanged` event fired for unrelated navigations. Clicking into Band Room for polls/pitches triggered the `pageChanged` subscriber which saw `_feedNavigatedFrom` as false (if GLStore wasn't subscribed yet when feed loaded) and removed the back bar.

**Fix:** Feed context now persists in `sessionStorage('gl_feed_context')`. The back bar survives:
- Page navigation within feed context (songs → detail, ideas → poll)
- Page refreshes (sessionStorage survives refresh)
- Only removed when: user clicks "Back to Feed", or navigates to the feed page

**Before:** Click poll from feed → navigate to Band Room → back bar disappears → no way back to feed
**After:** Click poll from feed → Band Room shows → back bar persists at top → 1-tap return

## Problem 2: "Needs Input" Showing After User Voted

**Root cause:** Poll items were classified as `needs_input` based solely on whether `voteCount < memberCount`. The current user's personal vote was never checked. So if Drew voted on all polls but Chris and Pierce hadn't, Drew still saw yellow "Needs Input" badges.

**Identity matching:** Poll votes are stored under display names (`"Drew"`, `"Chris"`) via `_bcGetName()` → `bandMembers[key].name`. New `_feedGetMyVoteKey()` replicates the same identity chain: `localStorage('deadcetera_current_user')` → `getCurrentMemberKey()` → `bandMembers[key].name` → `currentUserName` → email prefix.

**Fix:** Split "Needs Input" into two distinct categories:
- **Needs My Input** — polls I haven't voted on, ideas needing response
- **Needs Band Input** — polls I voted on but others haven't

**Before:** "Needs Input (5)" even though Drew voted on all 5
**After:** "Needs My Input (0)" + "Needs Band Input (5)" — or "All clear" in attention bar

## Problem 3: Practice Cards Showing Removed Songs

**Root cause (also fixed this session):** `_computeWeakSongs()`, `_renderBandReadinessSnapshot()`, `_ccRenderReadinessRadar()`, and the "What to Practice" weakest-songs list all iterated `allSongs` or `readinessCache` without checking `statusCache`. Songs with empty status (removed) or no status still appeared.

**Fix:** All four functions now filter to active statuses only: `prospect`, `learning`, `rotation`, `gig_ready`.

# GrooveLinx — Setlist Player Race Condition Fix

_Generated by Claude · 2026-03-24_

## Bug Report

- Open Grizz Fest setlist → click headphones
- Header shows "Wings Cafe 06/15/24" (stale prior session)
- Hangs on "Opening best version..."
- YouTube/Archive open external search pages
- Spotify button does nothing

## Root Cause

**Race condition on shared closure state in SetlistPlayer.**

1. Auto-resume fires on DOMContentLoaded if `gl_player_state` in localStorage is < 2 hours old
2. `_resumeFromState()` is async — loads setlists from Firebase, then calls `launch(wingsSetlist, "Wings Cafe")`
3. While that's awaiting `_buildQueue()` (sequential network calls per song), user clicks headphones on Grizz Fest
4. Second `launch()` runs concurrently — both mutate the **same** closure variables: `_queue`, `_setlistName`, `_currentIdx`
5. `_buildQueue()` pushes songs incrementally to shared `_queue` array — two concurrent builds corrupt it
6. Whichever `_createOverlay()` fires first wins — with potentially stale `_setlistName`

**Secondary bugs:**
- `_showFallback()` sets "Opening best version..." text that **never clears** — no timeout, no terminal state
- `close()` doesn't reset `_queue`, `_setlistName`, `_setlistId` — stale state persists across sessions
- `_openSpotify()` silently no-ops if `ListeningBundles.resolveUrl()` returns null and popup blocker kills `window.open`

## Fix: Launch Token Guard + Full Reset

### 1. Launch Token (`_launchToken`)
New monotonic counter incremented on every `launch()` call. After each `await`, the token is checked — if a newer launch superseded, the stale async operation **bails immediately**.

```
launch(Grizz):  myToken = 2 (++_launchToken)
  → fullClose() kills Wings Cafe session
  → await _buildQueue(Grizz, token=2)
    [each iteration checks: token !== _launchToken? → bail]
```

### 2. Full Reset on Launch
`launch()` now calls `fullClose()` + `clearResumeState()` before doing anything. This kills:
- Previous YouTube player instance
- Previous overlay DOM
- Now-playing bar
- `_queue`, `_currentIdx`, `_setlistId`, `_setlistName`
- localStorage resume state

### 3. Build Queue Isolation
`_buildQueue()` now builds into a **local array** and returns it. Only assigned to `_queue` after the build completes AND the token check passes. No more incremental mutation of shared state during async resolution.

### 4. Fallback Terminal State
"Opening best version..." replaced with: **"Couldn't load embedded version — choose a platform below"**
Platform buttons (YouTube/Archive/Spotify) always show. No infinite loading.

### 5. Search Timeout Tightened
Reduced from 10s to 5s. After timeout, immediately shows fallback with platform buttons.

### 6. Spotify Button Fix
`_openSpotify()` now:
- Wraps `ListeningBundles.resolveUrl()` in try/catch
- Always falls back to `open.spotify.com/search/{query}` — never silently no-ops
- Shows toast "Opening Spotify search..." on fallback

### 7. Resume Race Guard
`_resumeFromState()` snapshots `_launchToken` before its async work. If a user-initiated launch happens during Firebase load, resume bails.

### 8. Diagnostic Logging
Every launch logs: setlistId, setlistName, songCount, current song, artist, youtubeId, resolver state.

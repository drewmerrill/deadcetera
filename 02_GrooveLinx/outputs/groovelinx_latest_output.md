# GrooveLinx — Stabilization + Reliability

_Generated by Claude · 2026-03-24_

## Issues Found + Fixed

### 1. Race condition on rapid next/prev
**Issue**: `play()` used the queue-level `_token` for async guards. Rapid taps (next/next/next) would start multiple resolves sharing the same token — all would complete and overwrite each other.
**Fix**: `play()` now increments `_token` per-song-play. Each `_resolveAndPlay` call gets its own `myToken`. Stale resolves bail immediately.

### 2. No immediate visual feedback on tap
**Issue**: User taps a song → old embed stays visible while resolution runs in background. Feels broken.
**Fix**: `play()` immediately sets state to RESOLVING and emits `status: "Loading..."`. UI clears the video container and shows loading text before any async work starts.

### 3. Timeout math exceeded 4s budget
**Issue**: Resolver ran sources sequentially with 2s timeout each. 3 sources + 1 retry = 8s max inside a 4s hard timeout. The hard timeout caught it but UX was laggy.
**Fix**: Progressive timeout tightening — first source gets 1.5s, second gets 1s, third gets 0.8s. Total worst case: 1.5s + 1.5s (retry) + 1s + 0.8s = 4.8s. Hard timeout at 4s catches edge cases.

### 4. YouTube error didn't reset play state
**Issue**: `onError` callback triggered fallback but didn't set `_isPlaying = false`. Play/pause button showed wrong state.
**Fix**: `_isPlaying = false` set in error handler. Added console warning for debugging.

### 5. Stale embed visible during song transition
**Issue**: When switching songs, the previous YouTube/Spotify/Archive embed stayed in the video container until the new source resolved (could be seconds).
**Fix**: `_renderSong()` immediately clears the video container with "Loading..." text on every song change.

### 6. `retryCurrentSong` had no visual feedback
**Issue**: User taps Retry → nothing visible happens while resolution runs.
**Fix**: Emits `status: "Retrying..."` immediately before calling `play()`.

### 7. Resolver errors could crash silently
**Issue**: If `R.resolve()` threw an unexpected error, the engine would hang — no timeout, no fallback.
**Fix**: Wrapped entire resolve call in try/catch. On error → FALLBACK state with reason.

## Timing Budget (4s max)

```
Cached song:      [0ms]  → instant embed
YouTube (1st):    [0-1.5s] → parallel 7 backends
YouTube (retry):  [0-1.5s] → only if 1st source
Spotify:          [0-1s]   → cache or API
Archive:          [0-0.8s] → API search
Hard timeout:     [4s]     → FALLBACK state guaranteed
```

## State Guarantees

| Entry | Always exits as |
|-------|----------------|
| `play()` | PLAYING or FALLBACK (within 4s) |
| `retryCurrentSong()` | PLAYING or FALLBACK (within 4s) |
| YouTube embed error | FALLBACK |
| Resolver exception | FALLBACK |
| Timeout | FALLBACK |
| Rapid next/prev | Only latest play survives, others bail |

**Never**: indefinite loading, silent failure, stuck state.

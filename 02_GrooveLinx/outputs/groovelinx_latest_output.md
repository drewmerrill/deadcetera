## Auto-Resolution Logic

When a song has no YouTube ID (not cached, not in saved versions):

1. **"Finding best version…"** shown in video container
2. `_resolveYouTubeId()` searches via Invidious public API
3. Query: `"{song title} {artist}"` (e.g., "Althea Grateful Dead")
4. Tries two Invidious instances (5s timeout each) for redundancy
5. Takes first video result
6. **Caches ID** in `gl_yt_id_cache` permanently
7. **Plays video** automatically — no user action needed

### Invidious API
- Public, no API key required
- Returns JSON with `videoId` field
- Two instances tried: `vid.puffyan.us`, `invidious.fdn.fr`
- Falls back gracefully if both fail

### Resolution Chain (complete)
1. Local cache (`gl_yt_id_cache`) — instant
2. Saved versions (Firebase `spotify_versions`) — YouTube platform match
3. Default North Star version — extract YouTube ID from URL
4. **Auto-search via Invidious** — NEW
5. Fallback UI with platform links

## Cache Behavior

- Auto-searched IDs stored in `gl_yt_id_cache` alongside manually resolved IDs
- Never re-searched once found
- Cache invalidation on YouTube error still works (removes failed ID)
- Next play of that song re-searches

## Edge Cases

- **Both Invidious instances down:** Falls through to fallback UI with YouTube/Spotify/Archive links
- **Invidious returns wrong video:** User can skip; cache invalidation on error clears it for next time
- **AbortSignal.timeout not supported:** 5s fetch timeout may not work in older Safari; the fetch will just take longer
- **Spotify fallback:** Always opens `open.spotify.com/search/{song}` if no resolved URL

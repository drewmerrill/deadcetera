GrooveLinx Governance Layer
Purpose: This folder is the strategic governance and continuity layer for GrooveLinx.
It is NOT:
	•	a scratchpad
	•	a duplicate task manager
	•	random notes
	•	implementation documentation
GitHub Projects manages execution.
Claude manages implementation.
This folder preserves:
	•	current truth
	•	strategic direction
	•	architecture decisions
	•	continuity between AI sessions
	•	operational alignment

File Roles
CURRENT_STATE.md
Defines:
	•	what GrooveLinx currently is
	•	major systems/features
	•	current operational model
Update when:
	•	platform capabilities materially change

CURRENT_PRIORITIES.md
Defines:
	•	active strategic priorities
	•	sequencing
	•	major focus areas
Update when:
	•	priorities shift
	•	major initiatives complete

ARCHITECTURE_DECISIONS.md
Defines:
	•	durable architectural decisions
	•	workflow decisions
	•	operational design choices
Update when:
	•	architecture direction changes

KNOWN_TECHNICAL_DEBT.md
Defines:
	•	acknowledged technical debt
	•	structural weaknesses
	•	operational risks
Update when:
	•	debt is discovered or resolved

ACTIVE_WORKSTREAMS.md
Defines:
	•	major active initiatives
	•	current operational lanes
Update when:
	•	workstreams begin/end

CURRENT_PHASE.md
Defines:
	•	the current GrooveLinx phase
	•	what the organization is trying to accomplish RIGHT NOW
Should stay short and highly current.

CLAUDE_HANDOFF.md
Defines:
	•	immediate implementation context
	•	current implementation focus
	•	latest execution guidance
This is tactical and changes frequently.

PROJECT_INDEX.md
Defines:
	•	high-level map of important folders/systems/docs
Purpose: help humans and AI systems navigate the repo efficiently.

Operational Rules
	1	GitHub Projects is the task system.
	2	This folder is NOT a duplicate task board.
	3	Chats are temporary working sessions.
	4	Repo files are institutional memory.
	5	Avoid duplicate truth across files.
	6	Keep governance docs concise and current.
	7	Prefer updating existing docs over creating new random docs.
